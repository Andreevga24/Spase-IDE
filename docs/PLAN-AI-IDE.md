# Spase-IDE: план разработки IDE с встроенным ИИ (LLM)

Документ служит **единым ориентиром** для последующих итераций: при любых задачах по проекту сначала сверяться с этим планом, обновлять статусы этапов и фиксировать принятые архитектурные решения внизу файла.

---

## 1. Цель и границы продукта

### 1.1 Цель

Создать **настольную (или гибридную) IDE** с глубокой интеграцией LLM: чат, правки кода по запросу, контекст рабочей области, опционально — агентные сценарии (несколько шагов, терминал, поиск по коду), на уровне ожиданий, сопоставимых с **Cursor** (без копирования закрытых частей их продукта — только функциональные аналоги).

### 1.2 Что входит в MVP (минимально жизнеспособный продукт)

- Редактор кода с подсветкой синтаксиса и базовой навигацией по файлам.
- Открытие папки как «workspace».
- Панель **чата с LLM** с потоковым ответом (streaming).
- Передача в модель **контекста**: активный файл, выделение, список открытых файлов (ограниченно по размеру).
- Команда вида **«применить предложение к файлу»** (diff / patch), с подтверждением пользователем.
- Настройки: **API-ключ**, выбор **провайдера** (OpenAI-совместимый, Anthropic и т.д. — по мере реализации), базовые параметры модели.

### 1.3 Что отложить после MVP

- Полноценный LSP для всех языков «из коробки».
- Собственный индекс кодовой базы уровня Cursor (семантический поиск, авто-контекст на всём репо) — можно эволюционно добавлять.
- Совместное редактирование, облачный sync, marketplace расширений.
- Глубокая отладка (debugger UI) — по желанию, отдельным этапом.

## 3. Архитектурное решение: база приложения

Нужно **явно выбрать** один путь (зафиксировать в разделе 12 после решения):

### Вариант A — форк / расширение экосистемы VS Code (рекомендуется для скорости)

- **Electron + Code OSS** или разработка как **расширения** под VS Code / VSCodium.
- Плюсы: готовый редактор, терминал, Git, расширения, LSP.
- Минусы: тяжёлый рантайм, лицензионные нюансы при использовании бренда Microsoft.

### Вариант B — собственный shell на Tauri + Monaco / CodeMirror

- Плюсы: меньше размер, контроль UI/UX.
- Минусы: больше работы на паритет с «настоящей» IDE (Git, терминал, LSP).

### Вариант C — Web IDE (браузер) + бэкенд

- Плюсы: дистрибуция через URL.
- Минусы: файловая система и безопасность сложнее; для «как Cursor» чаще хотят desktop.

**Рекомендация для старта:** Вариант A или B в зависимости от команды; для соло-разработчика чаще выигрывает A (скорость) или B (бренд и лёгкость).

## 4. Архитектура ИИ-слоя

### 4.1 Компоненты

1. **Chat / Agent UI** — вкладки, история сессий, ввод, отмена генерации.
2. **Orchestrator** — сбор контекста, вызов LLM API, парсинг tool-calls (если есть).
3. **Context builder** — сбор: путь репо, git branch (опционально), выбранные файлы, user rules, лимиты токенов.
4. **Model adapters** — единый интерфейс: `streamChat`, `countTokens` (оценка), обработка ошибок, retry.
5. **Patch engine** — применение предложенных изменений к буферу редактора / на диск (с бэкапом и откатом).
6. **Telemetry (опционально)** — локальные логи, без отправки кода на сторонние серверы без согласия.

### 4.2 Провайдеры LLM (поэтапно)

- **Фаза 1:** OpenAI-совместимый HTTP API (многие прокси и open-weight сервера совместимы).
- **Фаза 2:** Нативный SDK/API Anthropic (если нужен отдельный протокол).
- **Фаза 3:** Локальные модели (Ollama, llama.cpp) — через тот же адаптер «OpenAI-compatible» где возможно.

### 4.3 Безопасность и приватность

- API-ключи: **только** в OS keychain / зашифрованное хранилище, не в репозитории.
- Явное предупреждение при отправке **содержимого файлов** наружу.
- Режим «не отправлять имена файлов / пути» (опционально, усложняет контекст).
- Для агента: **allowlist** команд или подтверждение каждой команды.

## 8. Критерии готовности (Definition of Done)

**MVP считается готовым**, если:

1. Пользователь открывает папку, редактирует файлы, сохраняет.
2. Пользователь ведёт диалог с моделью с streaming.
3. Содержимое активного файла/выделения реально попадает в запрос (проверяемо логом «debug prompt» в dev-режиме).
4. Пользователь может применить предложенные изменения с шагом подтверждения и без безусловной перезаписи без осознанного действия.
